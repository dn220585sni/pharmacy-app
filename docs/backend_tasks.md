## 1. Контекст

Ми розробляємо нову касу для аптечної мережі. Це Android-додаток (Flutter), який працює на стаціонарних комп'ютерах з Android OS, клавіатурою, мишею, USB-сканером штрихкодів і принтером.

Зараз додаток працює на mock-даних (демо-режим). Для переходу в production потрібні серверні сервіси, описані в цьому документі.

**Всі нові сервіси повинні бути зворотно сумісними** — існуючі клієнти (стара каса, постамат) не повинні зламатися.

**Протокол:** всі запити — `GET` через CSP. Всі відповіді — JSON:
```json
// Успіх:
{"Status": "OK", "Result": "...", ...дані...}

// Помилка:
{"Status": "BAD", "Result": "текст помилки"}
```

## 3. Відомі проблеми які потрібно виправити

Ці проблеми були виявлені при інтеграції і задокументовані в `server_changes.txt`.

### Проблема 1: Login і Logout не працюють

**Рутина:** KabService3, функція Opt
**Блоки:** `mode="login"`, `mode="logout"`
**Проблема:** Змінна `allok` ніколи не встановлюється в `1` — завжди повертає `"BAD"`.
**Вплив:** Нова каса не може авторизувати фармацевта.
**Пріоритет:** 🔴 Блокер — див. Задачу 6.

### Проблема 2: SpisanieBonusov працює тільки для постамату

**Рутина:** KabService2
**Проблема:** Сервіс працює з WZakaz (веб-замовлення), а не з клієнтськими кодами ЛАЙК.
**Вплив:** Для роздрібного продажу бонуси потрібно списувати через зовнішній API ЛАЙК.
**Пріоритет:** 🟠 Важливо — див. Задачу 7.

#### Навіщо

Коли фармацевт вибирає препарат у таблиці, справа з'являється детальна панель з інформацією: форма випуску, МНН, дозування, умови зберігання, термін придатності, фото, кількість в упаковці, попередження при вживанні. Зараз ці дані — mock. Потрібен серверний метод.

#### Запит

```
GET /csp/user/Kab.Service.cls?ServiceName=GetSKUdetail&ids={ids}
```

| Параметр | Тип | Обов'язковий | Опис |
|----------|-----|-------------|------|
| `ids` | String | Так | Код довідника товару (ukod). Формат: "5511*3*14" |

#### Відповідь (успіх)

```json
{
  "Status": "OK",
  "Result": "дані по товару",
  "ids": "5511*3*14",
  "name": "ЦИТРАМОН-Д №6",
  "manufacturer": "Дарниця",
  "category": "Знеболюючі",
  "inn": "Ацетилсаліцилова кислота + Парацетамол + Кофеїн",
```

#### Опис полів відповіді

| Поле | Тип | Обов'язкове | Опис | Звідки в Caché |
|------|-----|-------------|------|----------------|
| `ids` | String | Так | Код довідника | Параметр запиту |
| `name` | String | Так | Назва товару | `^Farm("Spr","SPR",1,ukod)` поле 1 |
| `manufacturer` | String | Так | Виробник | `^Farm("Spr","PROIZ",1,...)` |
| `category` | String | Ні | Категорія ("Знеболюючі", "Антибіотики", etc.) | Довідник категорій або "" |
| `inn` | String | Ні | МНН (міжнародна непатентована назва) | Довідник МНН або "" |
| `dosageForm` | String | Ні | Форма випуску ("Таблетки", "Капсули", "Сироп") | Довідник або "" |
| `dosage` | String | Ні | Дозування ("400 мг", "500 мг/5 мл") | Довідник або "" |
| `requiresPrescription` | String | Ні | Рецептурний: "1" = так, "0" = ні | Довідник або "0" |
| `expiryDate` | String | Ні | Термін придатності "MM/YY" (найближчий серед партій) | З `^Farm("Sklad",1,skod)` |
| `unitsPerPackage` | String | Ні | Одиниць в упаковці (для блістерного продажу). "" = не ділиться | Довідник або "" |
| `pharmacistBonus` | String | Ні | Бонус фармацевта (ціле число). "" = нема бонусу | Маркетинг або "" |
| `barcode` | String | Ні | Штрихкод EAN-13 | `^BarCode` |
| `series` | String | Ні | Серія (перша партія) | `^Farm("Sklad",1,skod)` |
| `serialNumber` | String | Ні | Серійний номер | Або "" |
| `storageConditions` | String | Ні | Умови зберігання | Довідник або "" |
| `isOwnBrand` | String | Ні | Власна торгова марка: "1" / "0" | Довідник або "0" |
| `analogueGroup` | String | Ні | Група аналогів (МНН або ATC-код) | Довідник або "" |
| `imageUrl` | String | Ні | URL фото товару | CDN або "" |
| `intakeWarning` | String | Ні | Попередження при вживанні | Довідник або "" |
| `availabilityStatus` | String | Ні | Для товарів з залишком 0: "marketShortage" / "quarantined" / "inTransit" / "awaitingReceiving" / "notOrdered" / "" | Логіка за даними складу |
| `storageLocations` | Array | Ні | Місця зберігання з кількостями | `stelazh`, `vitrina`, `polka`, `robot` з GetSKUprice |

#### Відповідь (помилка)

```json
{"Status": "BAD", "Result": "товар не найден"}
```

#### Примітки

- Якщо поле невідоме або відсутнє в Caché — повертайте **порожній рядок** `""`, а не пропускайте поле. Клієнт обробляє порожні рядки як "дані відсутні".
- Поле `expiryDate` — формат строго `"MM/YY"` (наприклад `"09/27"`). Якщо є кілька партій — повернути найближчий термін.
- Поле `storageLocations` — масив. Якщо товар лежить на стелажі (qty=8) і в роботі (qty=2), повернути обидва. Якщо тільки одне місце — масив з одним елементом. Якщо невідомо — порожній масив `[]`.
- Значення `type` в `storageLocations`: `"shelf"` (стелаж), `"showcase"` (вітрина), `"polka"` (полка), `"robot"` (робот).
- **Цей метод НЕ повертає ціну і залишок** — вони вже є в `GetSKUprice`. Клієнт зробить два паралельних запити.

#### Підказка де шукати дані в Caché

Більшість полів — в довідниках `^Farm("Spr",...)`. Якщо якогось довідника немає — поверніть "". Ми на клієнті покажемо "—" або сховаємо блок. Краще повернути пусте поле ніж не запустити сервіс.

#### Перевірка

```
http://localhost:57772/csp/user/Kab.Service.cls?ServiceName=GetSKUdetail&ids=5511*3*14
```

### Задача 2: Розширення SearchByName — додаткові поля

**Пріоритет:** 🟠 Важливо (покращить UX таблиці, але не блокер)

#### Навіщо

Зараз `SearchByName` повертає 6 полів: ids, name, manufacturer, shelf, qty, price. Для таблиці каси потрібно ще кілька полів щоб показувати бейджі (термін придатності, бонус, рецептурність) без додаткового запиту на кожен рядок.

#### Що додати

Розширити JSON кожного елемента в `items[]`:

```json
{
  "ids": "5511*3*14",
  "name": "ЦИТРАМОН-Д №6",
  "manufacturer": "Дарниця",
  "shelf": "70204",
  "qty": "10",
  "price": "16.60",

  "expiryDate": "09/27",
  "pharmacistBonus": "5",
  "requiresPrescription": "0",
  "unitsPerPackage": "6",
  "barcode": "4823006401024"
}
```

| Нове поле | Тип | Опис | Звідки |
|-----------|-----|------|--------|
| `expiryDate` | String | `"MM/YY"` або `""` | Найближчий термін серед партій |
| `pharmacistBonus` | String | Число або `""` | Маркетинг |
| `requiresPrescription` | String | `"1"` / `"0"` | Довідник |
| `unitsPerPackage` | String | Число або `""` | Довідник |
| `barcode` | String | EAN-13 або `""` | `^BarCode` |

#### Зміни в коді

В рутині KabService3, функція Price, блок `mode="searchbyname"`:

Після рядка:
```
..set return(i)="""price"":"""_fprice_"""}"
```

Вставити перед закриваючою `}`:
```
..set return(i)="""price"":"""_fprice_""","
..;; нові поля
..s bcode="" s:$d(^BarCode("ukod",1,ukod)) bcode=$o(^BarCode("ukod",1,ukod,""))
..s expDt="" ;; TODO: визначити найближчий expiryDate з партій
..s bonus="" ;; TODO: визначити pharmacistBonus з маркетингу
..s rxFlag="0" ;; TODO: визначити рецептурність
..s unPkg="" ;; TODO: визначити unitsPerPackage
..set return(i+1)="""expiryDate"":"""_expDt_""","
..set return(i+2)="""pharmacistBonus"":"""_bonus_""","
..set return(i+3)="""requiresPrescription"":"""_rxFlag_""","
..set return(i+4)="""unitsPerPackage"":"""_unPkg_""","
..set return(i+5)="""barcode"":"""_bcode_"""}"
..s i=i+5
```

> **Примітка:** Я не знаю точну структуру глобалів де лежать ці дані. Підставте правильні джерела. Якщо даних немає — повертайте порожній рядок.

#### Зворотна сумісність

Додавання нових полів НЕ зламає старих клієнтів — вони просто ігнорують невідомі поля в JSON.

#### Перевірка


### Задача 3: ProcessSale — реєстрація продажу

**Пріоритет:** 🔴 Блокер (без цього каса не фіксує продажі)

#### Навіщо

Зараз при натисканні "Оплата" каса очищує кошик, але **нікуди не записує продаж**. Потрібен сервіс, який:
1. Створює запис продажу в базі
2. Повертає номер чеку і ID транзакції
3. Опційно списує бонуси ЛАЙК (атомарно з продажем)

#### Запит

```
GET /csp/user/Kab.Service.cls?ServiceName=ProcessSale
  &user={фармацевт}
  &items={JSON-масив товарів}
  &paymentMethod={card|cash|mixed}
  &totalAmount={сума}
  &cashAmount={готівка}
  &cardAmount={картка}
  &bonusAmount={бонуси}
  &loyaltyPhone={телефон ЛАЙК}
  &discountPercent={відсоток знижки}
```

| Параметр | Тип | Обов'язковий | Опис |
|----------|-----|-------------|------|
| `user` | String | Так | Ім'я фармацевта (з GetUsers) |
| `items` | String (JSON) | Так | URL-encoded JSON масив (див. нижче) |
| `paymentMethod` | String | Так | `"card"`, `"cash"`, або `"mixed"` |
| `totalAmount` | String | Так | Фінальна сума з урахуванням знижок, формат `"123.45"` |
| `cashAmount` | String | Ні | Сума готівкою (для cash/mixed), `"0.00"` якщо картка |
| `cardAmount` | String | Ні | Сума карткою (для card/mixed), `"0.00"` якщо готівка |
| `bonusAmount` | String | Ні | Сума списаних бонусів ЛАЙК, `"0.00"` якщо нема |
| `loyaltyPhone` | String | Ні | Телефон клієнта ЛАЙК (формат `+380XXXXXXXXX`), `""` якщо нема |
| `discountPercent` | String | Ні | Персональна знижка (`"5"` = 5%), `"0"` якщо нема |

#### Формат items

```json
[
  {
    "ids": "5511*3*14",
    "name": "ЦИТРАМОН-Д №6",
    "qty": "2",
    "unitPrice": "16.60",
    "total": "33.20",
    "fractional": "0",
    "unitsPerPackage": ""
  },
  {
    "ids": "7890*1*5",
    "name": "НУРОФЕН ЕКСПРЕС 400мг",
    "qty": "1",
    "unitPrice": "89.50",
    "total": "89.50",
    "fractional": "3",
    "unitsPerPackage": "10"
  }
]
```

| Поле item | Опис |
|-----------|------|
| `ids` | Код довідника |
| `name` | Назва (для логу/чеку) |
| `qty` | Кількість упаковок (ціле число). Для дробового продажу = "0" |
| `unitPrice` | Ціна за одиницю |
| `total` | Сума за позицію |
| `fractional` | Кількість блістерів (якщо дробовий продаж). "0" якщо цілі упаковки |
| `unitsPerPackage` | Одиниць в упаковці (для дробового). "" якщо цілі упаковки |

#### Відповідь (успіх)

```json
{
  "Status": "OK",
  "Result": "Продаж зареєстровано",
  "transactionId": "TXN-20260314-001542",
  "receiptNumber": "Ч-000127",
  "timestamp": "2026-03-14 09:15:32"
}
```

| Поле | Опис |
|------|------|
| `transactionId` | Унікальний ID транзакції. Формат — на ваш розсуд, але має бути унікальним |
| `receiptNumber` | Номер чеку (для друку і відображення) |
| `timestamp` | Серверний час реєстрації |

#### Відповідь (помилка)

```json
{"Status": "BAD", "Result": "Товар 5511*3*14 відсутній на складі"}
```

#### Бізнес-логіка

1. **Перевірити залишки:** для кожного item — чи є на складі достатня кількість. Якщо ні — повернути помилку (не виконувати продаж частково).
2. **Списати зі складу:** зменшити залишки по партіях (FIFO — найстаріша партія першою).
3. **Зареєструвати продаж:** створити запис зі всіма деталями (хто, що, коли, скільки, спосіб оплати).
4. **Списати бонуси (якщо є):** якщо `bonusAmount > 0` і `loyaltyPhone` заповнений — списати бонуси ЛАЙК. **Важливо:** списання бонусів і реєстрація продажу мають бути атомарними — або обидва або жодне.
5. **Повернути transactionId і receiptNumber.**

#### Примітки

- Якщо атомарність бонуси+продаж складно реалізувати одним методом — зробіть ProcessSale БЕЗ бонусів, а бонуси клієнт спише окремим викликом (SpisanieBonusov). Але тоді додайте параметр `transactionId` в SpisanieBonusov для зв'язки.
- Формат `transactionId` — на ваш розсуд. Рекомендація: `TXN-{дата}-{порядковий}` або UUID.
- Формат `receiptNumber` — згідно з вашою нумерацією чеків.

#### Перевірка

### Задача 6: Виправити Login / Logout

**Пріоритет:** 🔴 Блокер

#### Навіщо

Фармацевт має авторизуватися при відкритті зміни. Зараз Login завжди повертає "BAD".

#### Поточна проблема

В KabService3, функція Opt, блоки `mode="login"` і `mode="logout"`: змінна `allok` ніколи не встановлюється в `1`.

#### Потрібно

**Варіант А (мінімальний, рекомендований для старту):**

Login перевіряє user+password зі списку GetUsers і повертає:

```json
// Успіх:
{
  "Status": "OK",
  "Result": "авторизовано",
  "user": "Микола",
  "token": "SESSION-ABC123DEF456"
}

// Помилка:
{"Status": "BAD", "Result": "Невірний логін або пароль"}
```

Де `token` — згенерований ідентифікатор сесії (зберігається в ^Session або аналог). Цей токен клієнт буде передавати в наступних запитах замість пароля.

Logout інвалідує токен:
```
GET ?ServiceName=Logout&token=SESSION-ABC123DEF456
```
```json
{"Status": "OK", "Result": "сесію завершено"}
```

**Варіант Б (якщо токени складно):**

Login просто перевіряє user+password і повертає OK/BAD. Без токена. Клієнт зберігатиме логін локально.

```json
{"Status": "OK", "Result": "авторизовано", "user": "Микола", "ipn": "1234567890"}
```

#### Перевірка

```
http://localhost:57772/csp/user/Kab.Service.cls?ServiceName=Login&user=Микола&pswd=1234
```

## 6. Загальні вимоги до всіх сервісів

### Формат відповіді

Всі сервіси повинні повертати JSON у форматі:
```json
{"Status": "OK|BAD", "Result": "текст", ...дані...}
```

### Кодування

windows-1251 або UTF-8 (згідно з поточним налаштуванням `^NewSPR("Service","SetCharSet")`). Клієнт підтримує обидва.

### Обробка помилок

- Невідомий ServiceName → `{"Status": "BAD", "Result": "unknown service"}`
- Обов'язковий параметр пустий → `{"Status": "BAD", "Result": "параметр X обов'язковий"}`
- Товар не знайдено → `{"Status": "BAD", "Result": "товар не найден"}`
- Внутрішня помилка → `{"Status": "BAD", "Result": "внутрішня помилка сервера"}` (не показувати stack trace)

### Числа

- Ціни: `"16.60"` (крапка як роздільник, 2 знаки після крапки)
- Кількості: `"10"` (ціле число як рядок)
- Бонуси: `"5"` (ціле число як рядок)
- Прапорці: `"0"` або `"1"`

### Порожні значення

Якщо поле невідоме — повертайте **порожній рядок** `""`, а не `null` і не пропускайте поле.

### Маршрутизація

Кожен новий сервіс додається в $Case таблицю в KabService.mac (або відповідній рутині). Формат:
```
"ServiceName":FunctionName^RoutineName("mode",.params,.return),
```

### Зворотна сумісність

**Жоден існуючий сервіс не повинен зламатися.** Нові параметри — завжди необов'язкові. Нові поля у відповіді — доповнюють, не змінюють існуючі.

### Тестування

Після реалізації кожного сервісу — перевірити через браузер на тестовому сервері:
```
http://10.90.77.66:57772/csp/user/Kab.Service.cls?ServiceName=XXX&param1=val1
```

Переконатися що:
1. Успішний запит повертає правильні дані
2. Запит з невалідними параметрами повертає `"BAD"` з зрозумілим текстом
3. Існуючі сервіси (GetSKU, GetSKUprice, SearchByName, GetUsers) все ще працюють
